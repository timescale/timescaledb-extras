# cagg_manual_refresh_ranges.sql

Generate refresh commands for continuous aggregate ranges that were left behind by a previously failed manual refresh or by a refresh policy job.

When a continuous aggregate refreshes data, TimescaleDB tracks the time ranges in the internal catalog 
table `_timescaledb_catalog.continuous_aggs_materialization_ranges`. This script
provides a view and a function to inspect those ranges and produce ready-to-run
refresh statements — without executing anything automatically.

(Note: the catalog table contains ranges that are being actively processed as well. So the function only generates CALL commands that are meant to be executed manually).

## Setup

Load the script in `psql`:

```sql
\i cagg_manual_refresh_ranges.sql
```

This creates the view and function. Re-run to update.

## Usage

### See all pending ranges
### NOTE: shows ranges that are being ACTIVELY processed and/or were leftover by a previous failure.
```sql
SELECT * FROM timescaledb_information.cagg_pending_ranges;
```

Example output:

```
       cagg_name        |  lowest_modified_value  | greatest_modified_value
------------------------+-------------------------+-------------------------
 public.sensor_hourly   | 2025-01-01 00:00:00+00  | 2025-01-02 00:00:00+00
 public.sensor_hourly   | 2025-03-10 00:00:00+00  | 2025-03-11 00:00:00+00
 public.daily_sales     | 2025-02-15              | 2025-02-16
```

### Generate refresh commands for a specific continuous aggregate

```sql
SELECT * FROM cagg_get_manual_refresh_stmt('public.sensor_hourly');
```
Sample output:
```
-[ RECORD 1 ]---+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
refresh_command | CALL refresh_continuous_aggregate('public.sensor_hourly_avg', '2025-12-31 19:00:00-05'::timestamptz, '2026-01-02 19:00:00-05'::timestamptz);
delete_command  | DELETE FROM _timescaledb_catalog.continuous_aggs_materialization_ranges WHERE materialization_id = 2 AND lowest_modified_value = 1767225600000000 AND greatest_modified_value = 1767398400000000;
-[ RECORD 2 ]---+--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
refresh_command | CALL refresh_continuous_aggregate('public.sensor_hourly_avg', '2026-02-28 19:00:00-05'::timestamptz, '2026-03-04 19:00:00-05'::timestamptz);
delete_command  | DELETE FROM _timescaledb_catalog.continuous_aggs_materialization_ranges WHERE materialization_id = 2 AND lowest_modified_value = 1772323200000000 AND greatest_modified_value = 1772668800000000;
```

Copy the `refresh_command` output and run it:

```sql
CALL refresh_continuous_aggregate('public.sensor_hourly',
    '2025-01-01 00:00:00+00'::timestamptz,
    '2025-01-02 00:00:00+00'::timestamptz);
```

Repeat for each range as needed. The `cagg_get_manual_refresh_stmt` function does **not** execute anything — it
only generates statements for you to review and run.

### Error messages:
```
CALL refresh_continuous_aggregate('public.sensor_hourly_avg', '2026-03-01 00:00:00+00'::timestamptz, '2026-03-05 00:00:00+00'::timestamptz);
psql:queries.sql:223: ERROR:  materialization range [2026-03-01 00:00:00+00, 2026-03-01 21:00:00+00) overlaps with existing range [2026-03-01 00:00:00+00, 2026-03-05 00:00:00+00) in materialization_ranges table for continuous aggregate "public"."sensor_hourly_avg"
DETAIL:  A concurrent refresh is working on this range or a previously failed refresh left an overlapping range in the materialization_ranges table.
```

If this happens, check if the range is being actively worked on by a manual refresh or policy. If not
run the DELETE statament generated by the function and then retry the manual refresh_continuous_aggregate call.

## Supported time column types

The script handles `timestamptz`, `timestamp`, `date`, and integer-based
partitioning columns automatically, generating correctly typed arguments in the
output.

## When to use this
- Troubleshooting for concurrent continuous aggregate refresh: checking if materialization ranges are accumulating
  unexpectedly.
